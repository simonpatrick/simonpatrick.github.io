The optimistation/tuning process

The process:

  • 理解系统架构（understand the system architecture）
  • 稳定系统(Statbilise the system)
  • 设置性能目标(Set the performance targets)
  • 衡量目前系统性能(Measure current performance)
  • 定位目前的瓶颈(Identify the current bottleneck)
  • 修正瓶颈的根本原因(Fix the root cause of the bottlenect)
  • Repeat util you meet the target

常见错误(common errors)

  • 优化不需要的代码
  • 无效测试(Insufficient testing)
  • realistic data volumes
  • realistic user load
  • 缺少清晰的目标(Lack of clear performance targets)
  • 猜测瓶颈在哪里(Guessing where the bottleneck is)
  • 修正现象而不是根源(Fixing the symptom rather than the cause)

Tomcat Tuning

Logging 简述

默认的配置是通用配置，可能有些配置不适合production

  • catch-all logger logs to file and stdout
  • no overflow protection
  • logging is synchronised(同步日志，一般需要换成异步日志)

生产日志

  • remove duplicate logging(logging.properties)

To：

  • Synchronous logging/Asynchronous logging
  • Synchronous logging can become a bottlenect
  • Asynchronous logging,log queue use memory/limit queue size to avoid OOM error
  • fall back to synchronised logging and/or drop some logging message

Connectors 连接器

  • 需要先了解一下内容：
  • 应用使用的模式
  • 网络情况
  • TCP连接情况
  • HTTP transactions
  • HTTP Keep-alive情况
  • SSL(https)
  • Layer 4 or Layer 7
  • connection pools

对于connector来说，需要了解，什么样的connector：

  • Java Blocking IO
  • Oldest-most stable
  • JSSE based SSL
  • Native(APR)
  • Non-Blocking
  • Use OpenSSL
  • Java Non-Blocking IO
  • JSSE based SSL

connector使用场景(未必正确)：
|Requirement|Connector in preference order|
|——|——————————————————|
|Stability|BIO APR/NIO|
|SSL|APR NIO BIO|
|Low concurrency|BIO APR NIO|
|High concurrency/No Keey live|BIO APR NIO|
|High concurrency/Keep Live|APR NIO BIO|

为什么你要使用NIO connector?? NIO和BIO切换很简单

Connector Tuning

  • maxThreads
    • 最大并发请求数
    • for BIO，最大的open/active connections
    • 一般在200-800
    • 400是个不错的值
    • heavy CPU usage -> decrease
    • light CPU usage -> increase
  • maxKeepAliveRequests
    • typical values 1,100
    • maximum number of HTTP requests per TCP connections
    • set to 1 to disable keep alive
    • disable for BIO with very high concurrency,layer 4 load balance,no sll
    • enable for ssl,apr/nio,layer 7 load balancer
    • Note BIO connector automatically disables keep alive when concurrent connections
      reach 75% of max Threads
  • connection Timeout
    • typical value 3000
    • default 20000 is too high for production
    • keep alive 使用同样timeout
    • increase for slow client
    • increase for layer 7 load balancer with connection pools and keep alive on
    • decrease for faster time-outs

Content cache tuning

  • dynamic content is not cached
  • static content is cached
  • configured using the element
  • cacheMaxSize: 10240
  • cacheTTL: 5000
  • cacheMaxFileSize: 512
  • NIO/APR can use SEND_FILE

JVM Tuning

  • memory
    • JAVA Heap(Xmx,Xms) is not same as process heap, aim to set as low as possible
      set too high might cause long GC cycles
      • -XX:NewSize/-XX:NewRatio
        • set 25-33% of total java heap
        • Setting too high or too low leads to inefficient GC
    • process heap includes:
      • Java Heap
      • Permanet Generation
      • Threads stacks
      • Native code
      • Directly allocate memory
      • Code generation
      • Garbage collection
      • TCP buffers
  • Garbage collection:GC
  • Short lived objects never reach the Old Generation
  • Short lived objects cleaned up by short minor garbage collections
  • Long lived objects promoted to Old Generation
  • Long lived objects cleaned up by (rare) full garbage collection
  • GC Pause the application - regardless of GC algorithm
  • Pause can be from milliseconds to seconds
  • The pause will impact your response time
  • -XX:MaxGCPauseMillis -XX:MaxGCMinorPauseMillis
  • Many More options: http://blogs.sun.com/watt/resource/jvm-options-list.html
  • Concurrent Mark Sweep
    – -XX:+UseConcMarkSweepGC
    – Does not use survivor spaces
    – Can be forced to; not recommended

Scaling Tomcat

  • Load balancing - routing requests to multiple tomcat instances
  • Clustering - sharing state between tomcat instance for fail-over
  • Simplest Configuration
    • httpd
    • mod_proxy_http
  • considerations
    • state management
    • fail over
  • stateless
    • requests routed to tomcat instances purely on load balancing algorithm
    • http sessions will not work
  • Http session support
    • tomcat maintains http session
    • all requests for a session routed to same tomcat instance

Hints and Tips

  • load balancing/clustering : at least 3 tomcat instances
  • redeployment can expose memory leak issue
